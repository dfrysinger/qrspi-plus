# Q16 + Q18 + Q22 + Q27: Prior art (task types, summary blocks, prompt regression, research→design handoff)

## Summary

**TL;DR:** Across AI-assisted development and engineering-process literature, task-type differentiation typically appears as either change-class taxonomies (DORA's change types, Conventional Commits) or agent-mode switches (Cursor modes, Aider modes), while prompt-regression guarding splits between deterministic snapshot tests and statistical/LLM-as-judge eval suites. Source-authored summary blocks converge on a small set of shapes (TL;DR, abstract, executive summary, BLUF, structured front-matter), and agentic frameworks transfer research→design output most often via structured artifacts (plans, scratchpads, shared state graphs) rather than free-form prose.

**Key findings:**
- DORA and Conventional Commits document task/change differentiation along type axes (feat/fix/chore/docs) used to drive automation downstream (https://www.conventionalcommits.org/en/v1.0.0/, https://dora.dev/).
- Cursor and Aider both expose explicit task "modes" (Ask/Agent/Edit; ask/code/architect/help) that gate which tools and prompts apply (https://docs.cursor.com/chat/overview, https://aider.chat/docs/usage/modes.html).
- Summary-block shapes in technical writing cluster around IMRaD abstracts, BLUF, executive summary, and TL;DR; tradeoffs are on length, audience scanning behavior, and risk of duplication drift (https://en.wikipedia.org/wiki/IMRAD, https://en.wikipedia.org/wiki/BLUF_(communication), https://www.nngroup.com/articles/inverted-pyramid/).
- Prompt-regression guards documented in practitioner literature: snapshot/golden tests (promptfoo, LangSmith), LLM-as-judge eval, embedding similarity, and behavioral assertion suites; tradeoffs are between brittleness (snapshots) and noise (LLM-judge) (https://www.promptfoo.dev/docs/configuration/expected-outputs/, https://docs.smith.langchain.com/evaluation, https://github.com/openai/evals).
- Research→design handoff in agentic frameworks: LangGraph uses typed shared state; AutoGen passes messages through GroupChat; OpenDevin/SWE-agent use a planning agent that emits a plan artifact consumed by an executor; BAML emits typed structured outputs as the contract (https://langchain-ai.github.io/langgraph/concepts/low_level/, https://microsoft.github.io/autogen/, https://docs.all-hands.dev/, https://www.swe-agent.com/, https://docs.boundaryml.com/).

**Surprises:** Aider documents an explicit "architect" mode whose role is to produce a design artifact consumed by a separate "editor" model — a literal research→design→code split inside a single CLI (https://aider.chat/docs/usage/modes.html). Anthropic's published research on its multi-agent research system describes orchestrator→subagent handoff via written artifacts and explicit task descriptions, with subagents themselves further decomposing (https://www.anthropic.com/engineering/built-multi-agent-research-system).

**Caveats:** Quantitative evidence on cost/quality tradeoffs is sparse outside Anthropic's published numbers and a few academic papers; most framework docs describe mechanisms without measured outcomes. Some sources (Copilot Workspace technical details) are blog posts or limited preview docs.

## Full findings

### Q16: Task-type differentiation prior art

**Change-type taxonomies driving automation.** Conventional Commits formalizes a small set of commit types (feat, fix, chore, docs, refactor, test, build, ci, perf) used by downstream tools (semantic-release, changelog generators) to differentiate behavior per type — e.g., `feat` triggers a minor bump, `fix` a patch (https://www.conventionalcommits.org/en/v1.0.0/). Angular's commit message convention is the original source for this taxonomy (https://github.com/angular/angular/blob/main/contributing-docs/commit-message-guidelines.md).

**DORA classification.** DORA distinguishes change types and uses change failure rate per change as a core metric, but treats "change" as a unit rather than splitting it by task class; the governance pattern is measurement and feedback rather than per-type routing (https://dora.dev/guides/dora-quick-check/).

**Agent mode/role switches.** Cursor exposes Ask, Agent, and Edit modes with different tool access and prompts (https://docs.cursor.com/chat/overview). Aider documents `code`, `architect`, `ask`, and `help` modes with distinct system prompts and tool surfaces; `architect` produces design instructions consumed by a separate `editor` model in a two-step pattern (https://aider.chat/docs/usage/modes.html, https://aider.chat/2024/09/26/architect.html). GitHub Copilot Chat differentiates "agent mode" from "ask" and "edit" with explicit mode names (https://docs.github.com/en/copilot/using-github-copilot/asking-github-copilot-questions-in-your-ide).

**Issue/PR labeling and routing.** GitHub issue forms and label-based routing (e.g., `bug`, `enhancement`, `question`, `documentation`) are the canonical SCM-level task-type axis; CODEOWNERS and label-triggered Actions implement governance/correction patterns (https://docs.github.com/en/communities/using-templates-to-encourage-useful-issues-and-pull-requests/configuring-issue-templates-for-your-repository).

**SWE-bench / SWE-agent.** SWE-agent's Agent-Computer Interface differentiates subtasks (file navigation, edit, test) via tool affordances rather than declared task types (https://swe-agent.com/, https://arxiv.org/abs/2405.15793).

**Correction patterns.** Conventional Commits and semantic-release require revert/footer conventions (`BREAKING CHANGE:`, `Revert`) for correction (https://www.conventionalcommits.org/en/v1.0.0/#specification). DORA's correction pattern is mean time to restore (MTTR) measured at the change level (https://dora.dev/guides/dora-quick-check/).

### Q18: Source-authored summary block shapes

**IMRaD abstract.** Introduction–Methods–Results–and–Discussion structure with a structured abstract at the head; tradeoff is rigid section ordering vs. easy scanning by readers in the same field (https://en.wikipedia.org/wiki/IMRAD).

**BLUF (Bottom Line Up Front).** Originated in US military communication; the conclusion/recommendation is stated first, supporting detail follows. Tradeoff: optimal for decision-makers, weaker for readers who need the reasoning chain to evaluate the conclusion (https://en.wikipedia.org/wiki/BLUF_(communication)).

**Executive summary.** Common in business/consulting reports; longer than TL;DR (often 1–2 pages) and self-contained. Tradeoff: high duplication with body, high drift risk during edits (https://en.wikipedia.org/wiki/Executive_summary).

**TL;DR.** Net-native short summary, 1–3 sentences, typically informal. Low maintenance cost, low information density (https://en.wikipedia.org/wiki/TL;DR).

**Inverted pyramid (journalism).** Most important info first, broadening detail below; Nielsen Norman Group documents that web readers scan and benefit from this structure (https://www.nngroup.com/articles/inverted-pyramid/).

**Structured front-matter / abstracts in standards docs.** RFCs use an Abstract section followed by Status of This Memo and Table of Contents (https://www.rfc-editor.org/rfc/rfc7322.html). ADRs (Architecture Decision Records) typically open with Status, Context, Decision, Consequences (https://adr.github.io/).

**arXiv/academic abstracts.** Single paragraph, usually 150–300 words, structured implicitly (problem, method, result, implication); reader tradeoff is parseability per discipline conventions (https://arxiv.org/help/abstract).

**Tradeoffs across shapes:**
- TL;DR vs. executive summary: maintenance cost rises with length; longer summaries drift faster from body content.
- BLUF vs. IMRaD: BLUF is faster for decisions, weaker for evaluative readers who need methodology before accepting findings (https://en.wikipedia.org/wiki/BLUF_(communication)).
- Structured (key findings as bullets) vs. prose: bullets scan faster (https://www.nngroup.com/articles/how-users-read-on-the-web/) but lose connective reasoning.

### Q22: Prompt-regression guard patterns

**Snapshot/golden testing.** Promptfoo runs declarative test suites with `expected` outputs and assertion types (equals, contains, similar, llm-rubric); regressions are flagged when outputs no longer satisfy assertions (https://www.promptfoo.dev/docs/configuration/expected-outputs/). LangSmith offers dataset-based regression evaluators (https://docs.smith.langchain.com/evaluation/concepts). Tradeoff: snapshots are brittle to harmless wording changes — high false-positive rate on creative outputs.

**LLM-as-judge.** OpenAI Evals, LangSmith, and promptfoo all support LLM-graded assertions where a model scores output against criteria (https://github.com/openai/evals/blob/main/docs/build-eval.md, https://www.promptfoo.dev/docs/configuration/expected-outputs/model-graded/). Tradeoff: lower brittleness, higher noise; judge drift across model versions; costs scale with eval set size.

**Embedding similarity.** promptfoo's `similar` assertion uses cosine similarity against a reference embedding with a threshold (https://www.promptfoo.dev/docs/configuration/expected-outputs/similar/). Tradeoff: tolerates paraphrase but can pass semantically wrong outputs that are topically close.

**Behavioral / property-based assertions.** Guardrails AI and Pydantic-based output validators check structural properties (JSON schema, regex, value ranges) rather than exact text (https://github.com/guardrails-ai/guardrails). BAML enforces typed output schemas at the framework level (https://docs.boundaryml.com/guide/introduction/what-is-baml). Tradeoff: catches structural regressions cheaply, misses content regressions.

**Adversarial / red-team test sets.** Anthropic and OpenAI document holding out adversarial prompts to detect regression on safety properties (https://www.anthropic.com/news/evaluating-feature-steering, https://openai.com/index/our-approach-to-alignment-research/). Tradeoff: high signal on targeted properties, high authoring cost.

**A/B comparison vs. baseline.** promptfoo and LangSmith both offer side-by-side comparison runs across prompt versions (https://www.promptfoo.dev/docs/getting-started/, https://docs.smith.langchain.com/evaluation/how_to_guides/compare_experiment_results). Tradeoff: requires a stable eval set; comparing across model upgrades conflates prompt and model effects.

**Reported tradeoff axes (from practitioner write-ups):**
- Maintenance cost: snapshot > LLM-judge > schema validation (highest to lowest authoring/upkeep) — discussed in (https://hamel.dev/blog/posts/evals/) and (https://eugeneyan.com/writing/evals/).
- Signal quality: schema/property assertions are deterministic but narrow; LLM-judge is broader but noisier (https://hamel.dev/blog/posts/llm-judge/).
- False-positive rate: exact-match snapshots highest; embedding-similar middle; rubric-based LLM-judge variable, depends on rubric quality (https://www.promptfoo.dev/docs/configuration/expected-outputs/).

### Q27: Research → Design info transfer in agentic frameworks

**LangGraph: typed shared state.** Agents read and write to a typed `State` object passed through a directed graph; the research node's output schema is the design node's input schema, enforced by Python types (https://langchain-ai.github.io/langgraph/concepts/low_level/#state). Reported property: deterministic handoff, but state schema must be designed up front (https://langchain-ai.github.io/langgraph/tutorials/introduction/).

**AutoGen: message-based GroupChat.** Agents exchange natural-language messages in a shared chat with a GroupChatManager routing turns (https://microsoft.github.io/autogen/0.2/docs/topics/groupchat/). Reported property: flexible but token-heavy; the full chat history travels with each turn (https://microsoft.github.io/autogen/0.2/docs/Use-Cases/agent_chat).

**OpenDevin / OpenHands: planner→executor with persistent plan artifact.** A planner agent emits a plan structure consumed by executor agents; state and history persist in an event stream (https://docs.all-hands.dev/, https://arxiv.org/abs/2407.16741). The OpenDevin paper reports planner-executor decomposition improves task completion on SWE-bench Lite vs. monolithic agents (https://arxiv.org/abs/2407.16741).

**SWE-agent: tool-mediated handoff.** No explicit research/design split; the agent navigates and edits via an Agent-Computer Interface and accumulates context in its trajectory (https://arxiv.org/abs/2405.15793). Cost evidence: trajectories on SWE-bench average tens of thousands of tokens per task (Table 4 in the paper).

**Aider architect/editor split.** The `architect` model produces design instructions in natural language; the `editor` model receives those instructions plus repo context and emits diffs (https://aider.chat/2024/09/26/architect.html). Aider's blog reports the architect/editor split improves benchmark scores on the Aider polyglot benchmark over single-model runs but at roughly 2x cost (https://aider.chat/2024/09/26/architect.html).

**Anthropic multi-agent research system.** Lead orchestrator decomposes the question and spawns parallel subagents; each subagent receives a written task description and returns written findings; orchestrator synthesizes (https://www.anthropic.com/engineering/built-multi-agent-research-system). Reported evidence: ~90% improvement on internal research evals over single-agent Claude, with ~15x token usage (https://www.anthropic.com/engineering/built-multi-agent-research-system).

**BAML: typed function contracts.** Each LLM "function" declares input and output types; the design stage's output is the input type of the next stage, enforced by codegen (https://docs.boundaryml.com/guide/introduction/what-is-baml). No published quality numbers; the framework's claim is reduced parsing failures vs. ad-hoc JSON prompting (https://www.boundaryml.com/blog/type-safe-llms).

**GitHub Copilot Workspace.** Documents a four-stage pipeline: Specification → Plan → Implementation → Test/Iterate, with the user able to edit each artifact before downstream stages run (https://githubnext.com/projects/copilot-workspace). Each stage's output is a structured artifact (markdown spec, file-level plan) that becomes input to the next; no published quantitative cost/quality numbers.

**Cross-cutting evidence:**
- Structured/typed artifacts (LangGraph, BAML, Copilot Workspace) trade up-front schema work for deterministic downstream consumption.
- Free-form message handoff (AutoGen GroupChat) trades flexibility for token cost and ambiguity.
- Two-model architect/editor splits (Aider) report quality gains at ~2x cost (https://aider.chat/2024/09/26/architect.html); orchestrator/subagent splits (Anthropic) report larger gains at ~15x token cost (https://www.anthropic.com/engineering/built-multi-agent-research-system).
